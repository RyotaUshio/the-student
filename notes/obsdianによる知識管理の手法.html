Tags (タグ)

#Tags

タグの目的はなんだろうか？

  • 検索可能性 - あとからサクッと検索して情報を発見できる
  • 分類 - フォルダの拡張
    • 関連: TagFolder

細則
...

  • 前提として、タグは case-insensitive である。 したがって、大文字・小文字については、ケースバイケースで単に見栄えのよいほうを選択する。
  • タグ中では、単語と単語の間にはハイフンを用いる（kebab-case）こととし、camelCase や PascalCase、snake_case などは用いない。たとえば、#something-like-this のようにする。
    • 特に、PascalCase のような区切り文字を用いない方法は使うべきでない。たとえば、"AB-CDE" という固有名詞があったら対応するタグはどうなってしまうだろうか？
    • あとで snake_case に変更したくなるかもしれないが、Tag Wrangler がどうにかしてくれるだろう。
  • タグは properties ではなく、本文中に置く。Webpage HTML Export プラグインでエクスポートしたときにもタグが visible になるようにするため。
  • ノート全体に対するタグは、新規ノートへのコピーを容易にするため、ノートの先頭にまとめて置く。
    • リンクをクリックしてノートを作成する場合には TagFolder の "Create a new note with the same tags" コマンドが利用できないので、やはり必要な処置である。
    • 必ずしも 1 行でなくてもよい。
  • それ以外のタグは行頭または行末に置くことを原則とし、行の途中にタグを置くことは可能であれば避ける。別の場所にコピー・ペーストするときの障壁とならないようにするため。
    • しかし、リンクの場合と同じように、通常の文字のかわりにタグを使いたくなるのも事実である（たとえば Obsdianによる知識管理の手法^like-this）。このルールは破ったとしても代償はあまり大きくないので、そこまで厳格に守らなくてもよい。むしろ、このルールに拘泥して余分な作業を発生させることは避けるべきである。
  • #Tags/nested Nested tags は、タグ・概念どうしの上下関係を不必要に固定化し、タグの良いところを台無しにするおそれがあるため、真に包含関係が成立すると考えられる場合を除いて使わない。複数のタグによって代用する。
    • たとえば、nested tags は tags に真に包含されるため、nested tag #tags/nested の使用が許される。むしろ、#nested-tags といった独立したタグを使うべきではない（#tags で検索したときにヒットしないため）。
    • 一方で、Obsidian と PKM は共通部分をもつが、一方が他方に含まれるわけではないため、#obsidian/pkm や #pkm/obsididan は使わず、#obsidian と #pkm のふたつのタグを付与するべきである。
    • また、personal knowledge management と PKM は（少なくともタグとして用いる場合は）等価であると考えられるため、#personal-knowledge-management と #PKM が併存するべきではない。どちらか一方に統一するべきである。
      • なお、この例では #PKM のほうがよいだろうが、いつでも短いほうがよいタグであるとは限らない。これはプログラミングにおける変数の命名と同じことである。
  • #question たとえば、 #obsidian におけるタグと #zotero におけるタグは一致させるべきだろうか？

Headings (見出し)
...

#headings

  • #Inline-title vs h1 の論争があるが、当面は inline title を使う。
  • ノート内の heading の最大レベルは、h2/## とする。
    • Inline title を使うか、ノートタイトルと同一の h1 を使うかにかかわらず見やすいノートを作るため。
  • 最小レベルについては特に規則を定めないが、h5 や h6 が出てきたらノートの分割を検討するとよいかもしれない。あまりに細かい headings は、フォントサイズによってレベルを識別することを難しくする。
  • 色分けによって headings のレベルをわかりやすくする方法は、視覚的に余計な撹乱要素となるのみならず、色分けのルールが変更されたさいに混乱の原因となる可能性があるので、採用しない。
  • Outline による快適な navigation を可能とするため、ある程度の意味的なまとまりごとにきちんと heading を置くことが望ましい。
  • Heading 内には、原則としてリンクを含まない。
    • ファイル名の変更にともなって Obsidian がリンクを自動アップデートした場合に、そのリンクを含む heading へのリンクが破壊される可能性がある。 #links/break
    • また、コアプラグインの Outline はマークダウンのレンダリング機能をもたない。
    • 代替策としては、たとえば heading の直後にリンクを置くことが考えられる。

ノートの粒度
...

  • #Atomic-note だなんだとか浮かれて、なんでもかんでも個別のノートに分割することは推奨されない！

    • ちょっとした内容ごとに新規ノートを作成するのは、単純に疲れる。
    • あまりにノートが増えすぎると、vault 内の navigation が困難になる。
      • #Quick-Switcher やリンク補完で、余計な候補が多く表示されるようになり、ノイズが増える。目的のノートにたどりつくことが難しくなる。
      • これらの機能では、サジェスト対象のファイル数（おそらく vault 内のファイル数とは限らない）が 1 万を超えると fuzzy search が無効になってしまう。

  • ノートを分割するかどうかのひとつの判断基準として、リンクをクリックするコストと、スクリーンをスクロールするコストの大小を比較することが考えられる。

  • ひとつの概念ごとにひとつのノートを用意したくなるかもしれない。しかし：

    • ある概念に関する内容はひとつのノートにしか書き込めないということになると、無用なコストとストレスが発生する。書くときには余計なことを考えずに好き勝手なノートに書き込んで、あとから集約したり検索したりすることのできるシステムのほうが優れている。同様のことは PDFのメモ についてもいえる。

    • この目的は、 #tags によって代用できる可能性が高い。

    • タグのほうが劣ることがあるとすれば、リンクでは日本語が使えるのに対し、タグでは 英数字 およびハイフンしか使用できない点がある。

    • 以下の例（PDFの取り扱い > ❌GoodNotes）を考える：

      この例では、"GoodNotes" をリンク化し、"GoodNotes.md" を作成することもできる。しかし、そうすると GoodNotes に関する情報をすべてそのノートに集約したくなってしまい、いろんなノートに適当に書き散らすことを discourage する可能性があるため望ましくない。

      • ただし、"GoodNotes.md" をタグノート化すれば、この両者のいいとこどりができる可能性がある。
        • こうすれば、あとから方法論を変更したくなったときにも事故を起こさない。
        • しかし、タグ #goodnotes を クリックして検索したり、Tags view から検索をかけたりするのとなにが違うのだろうか？余計なノートを増やしすぎて、ノイズを発生させる結果にならないだろうか？

  • 同一ノート内の #headings や #blocks へのリンクでは、ノート名を省略して [[#heading]] あるいは [[#^block-id]] とすることができる。しかし、この方法ではノートの一部を個別のノートに切り出した場合にリンクの破壊 が発生する。また、 2023-12-31 現在、この問題はコアプラグインの Note Composer を用いてノートを分割した場合では起こる (https://forum.obsidian.md/t/note-composer-links-to-blocks-and-headers-should-be-updated-when-extracting/37534 ) 。したがって、いかなる場合でもリンクにはノート名を含まなければならない。 #links/break

どの程度リンクさせるか？
...

この話題は ノートの粒度 とも関わりが深い。

リンク化できる（[[...]] で囲える）可能性がある語をすべてリンク化することは、不必要な作業を発生させるのみならず、Backlinks や Outgoing links にノイズを増やす原因にもなるので、推奨されない。

以下のルールは試行段階であり、厳格に守らなければいけないものではないが、現時点でのベストプラクティスをまとめた結果である。

  • 同一のリンク化可能語が同一ノート内に複数回登場する場合は、各意味段落ごとに 1 回までしかリンク化しない。

また、概念ひとつごとにひとつのノートを用意したい誘惑にかられることがあるかもしれないが、タグを使ったほうが良い結果になることが多い。Obsdianによる知識管理の手法 > ^0da73e を参照。
